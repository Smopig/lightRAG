## 1. 文件刪除後知識圖譜不一定乾淨 — 刪除文件的正確 SOP

### 問題說明

LightRAG 以 chunk 為單位建立 entity / relation，多個文件常共享同一個 entity
（例如同一個系統名稱出現在多份 PDF）。直接刪除來源文件，若底層儲存沒有正確
做 reference-counting 或 graph 邊的回收，會留下：

- entity 仍存在但其所有 reference chunk 已不存在（孤兒節點）。
- relation 仍存在但對應的 source chunk 已刪除。
- 向量庫中殘留已刪除文件的 embedding。

### 建議做法

1. **先確認 LightRAG 版本是否支援 `/documents/{doc_id}` DELETE API**（用
   `--help` 或 OpenAPI `/docs` 確認，不要假設）。
2. 刪除前先用 `query_lightrag.py` 對該文件相關的關鍵 entity 做一次查詢，
   記錄目前的 reference 清單到 `outputs/` 暫存，作為刪除後比對基準。
3. 執行官方刪除 API（若有），刪除後重新查詢同一批 entity：
   - 若 entity 完全消失：正常。
   - 若 entity 仍存在但 `references` 變成空陣列：屬於孤兒節點，需人工確認
     是否該 entity 由其他文件也提及（可保留），或需手動清理。
4. 若版本不支援乾淨刪除（常見於早期版本），採取**重建策略**：
   - 將該來源文件從 `raw_sources/` 對應子集移出（不可刪除 raw_sources 本身，
     依規則 1，只能搬到 `raw_sources/_excluded/` 之類的標記目錄並記錄原因）。
   - 清空 `lightrag_workspace/`（停服務後），重新 ingest 剩餘文件。
   - 此策略成本高，僅在文件數量少、重建時間可接受時使用。
5. 刪除後務必跑 `python scripts/wiki_lint.py`，檢查 wiki 中是否有頁面引用
   了已刪除的 source，並更新 `wiki/sources/` 對應索引。
6. 所有刪除動作（含「為何刪除」「採用哪種策略」）寫入 `wiki/log.md`。

### 相關命令

```bash
# 確認是否有刪除 API
curl -s http://localhost:9621/openapi.json | python3 -c "import sys,json;d=json.load(sys.stdin);print([k for k in d['paths'] if 'document' in k])"

# 刪除文件（若支援）
curl -X DELETE http://localhost:9621/documents/<doc_id>

# 刪除後檢查孤兒 entity（範例：用查詢比對）
python scripts/query_lightrag.py "列出與 <關鍵字> 相關的 entity 與其 references"
```

### 風險

- 孤兒節點會污染後續查詢結果（出現「查得到 entity 但查不到來源」的矛盾）。
- 重建策略可能耗費大量 LLM token（見第 2 節成本估算）。
- 誤刪 `raw_sources/` 違反操作安全規則第 1 條，務必只搬移不刪除。

## 2. 匯入成本估算與煞車機制（LLM cache 遷移、批量上限）

### 問題說明

LightRAG 在 entity/relation 抽取階段會對每個 chunk 呼叫 LLM，文件量一大，
token 成本與時間成本會快速放大。若中途中斷或重跑，沒有 cache 會重複付費。

### 建議做法

1. **估算先行**：匯入前先用簡單腳本估算 chunk 數量與預期 LLM 呼叫次數：
   - chunk 數 ≈ 總字數 / chunk size（預設約 1200 tokens，依 `.env` 設定）。
   - 每個 chunk 約對應 1～2 次 entity/relation 抽取呼叫。
   - 用 `MAX_ASYNC_LLM` 估算並行度，推算總時間。
2. **批量上限**：不要一次把整個 `raw_sources/` 丟進去。建議：
   - 第一批先用 1～3 份代表性文件做 pilot，確認抽取品質與 prompt 設定正確。
   - 確認無誤後，分批（例如每批 10～20 份）匯入，每批結束後檢查
     `lightrag_workspace/` 的 log 與 `wiki_lint.py` 結果。
3. **LLM cache 遷移**：LightRAG 預設會把 LLM 回應快取在
   `lightrag_workspace/kv_store_llm_response_cache.json`（或類似檔名，依版本
   確認實際檔名）。
   - 換模型或換 API key 時，**不要直接刪除 cache**，先備份
     （`cp` 到 `outputs/` 或用 `backup_lab.sh`）。
   - 若新模型的輸出格式不同（例如 entity JSON schema 改變），舊 cache 可能
     導致格式不一致，此時才考慮清空對應 cache 並重跑。
4. **煞車機制（中斷保護）**：
   - 設定 `MAX_PARALLEL_INSERT=1`（已在 `.env.example` 中），避免多份文件
     同時寫入造成 graph 寫入衝突。
   - 用 `timeout` 包住長批次匯入指令，避免無限卡住：
     ```bash
     timeout 3600 lightrag-server ... # 視實際 ingest 指令調整
     ```
   - 監控 `outputs/` 或 log 中的錯誤率，若單批失敗率 > 20%，立即停止後續批次，
     先排查原因（通常是 API rate limit 或 prompt 格式問題）。

### 相關命令

```bash
# 估算文件總大小（粗略 token 估算：英文約 4 chars/token，中文約 1.5-2 chars/token）
find raw_sources -type f \( -name '*.pdf' -o -name '*.md' -o -name '*.txt' \) -exec wc -c {} +

# 備份 cache 後才動它
cp lightrag_workspace/kv_store_llm_response_cache.json outputs/cache_backup_$(date -u +%Y%m%dT%H%M%SZ).json
```

### 風險

- 估算只是粗略參考，實際成本依模型計價、語言、prompt 長度差異很大；
  正式大量匯入前務必先用小批量試跑並記錄實際花費到
  `docs/IMPLEMENTATION_NOTES.md`。
- 清空 cache 會導致下次 ingest 全部重新呼叫 LLM，成本等同重新匯入。

## 3. Wiki 知識是否回流到 LightRAG（第一版維持單向）

### 問題說明

理論上，Wiki 中人工補充、修正的知識（例如糾正了 LLM 抽取錯誤的 entity
關係）也可以再餵回 LightRAG，形成雙向迴路。但這會引入「LightRAG 抽取結果」
與「人工編輯結果」互相覆蓋、版本漂移、來源可信度混淆等問題。

### 第一版決策：維持單向（LightRAG → Wiki Compiler → wiki/）

理由：

1. **可驗證性優先**：Wiki 頁面的價值在於每個陳述都能追溯到
   `wiki/sources/` 中的原始 reference。若把人工編輯內容回灌 LightRAG 並重新
   抽取，會混淆「原始文件來源」與「人工合成內容」，破壞可驗證性。
2. **避免 feedback loop 污染**：LLM 抽取結果若混入 LLM 自己先前生成的 wiki
   內容，容易產生自我強化的錯誤（hallucination 被當作事實再抽取一次）。
3. **複雜度可控**：第一版的重點是把「能不能查到、查到的東西能不能信」做穩，
   雙向同步的衝突解決機制（merge、版本控制、衝突偵測）成本遠超第一版範圍。

### 未來作法（規劃方向，不在第一版實作）

- 若要回流，建議走「**獨立 corpus** 路線」：把 Wiki 中經人工審核、標記為
  `verified` 的頁面，整理成一份**新的、獨立的原始資料**（例如
  `raw_sources/wiki_curated/`），明確標註其來源是「人工整理」而非「原始文件」，
  再以正常 ingest 流程匯入 LightRAG 的另一個 namespace 或獨立 workspace。
- 這樣 LightRAG 中會同時存在「原始文件抽取的知識」與「人工整理後回灌的知識」，
  但兩者來源標籤清楚分離，查詢時可依需求過濾。
- 需要先有版本控制（git 對 wiki/ 本身已經是版本控制）與審核流程
  （哪些頁面允許回灌）才能啟動此功能。

### 風險

- 若使用者誤以為編輯 wiki/ 會自動更新 LightRAG 圖譜，會造成查詢結果與
  wiki 內容不一致的誤解；務必在 `wiki/index.md` 或文件中明確標示「目前為單向」。

## 4. 服務常駐化（systemd / launchd）

### 問題說明

開發階段常用前景指令啟動 `lightrag-server`，但長期使用需要開機自動啟動、
崩潰自動重啟、日誌集中管理。

### 建議做法

- Linux：使用 **systemd user unit**（範本見 `deploy/lightrag-server.service`），
  不需要 root 權限，隨使用者 session 啟停。
- macOS：使用 **launchd**（`~/Library/LaunchAgents/*.plist`），概念與
  systemd user unit 對應：`ExecStart` ↔ `ProgramArguments`，
  `EnvironmentFile` ↔ `EnvironmentVariables`（plist 不支援讀取 .env 檔，
  需用 wrapper shell script 自行 `source .env` 後再 exec）。

### 相關命令

```bash
# 安裝 user unit
mkdir -p ~/.config/systemd/user
cp deploy/lightrag-server.service ~/.config/systemd/user/
# 編輯其中的 WorkingDirectory 為實際絕對路徑
systemctl --user daemon-reload
systemctl --user enable --now lightrag-server.service
systemctl --user status lightrag-server.service
journalctl --user -u lightrag-server.service -f
```

### 風險

- user unit 預設在使用者登出後可能被終止，需要
  `loginctl enable-linger <user>` 才能在登出後持續運作（此為系統層設定，
  需使用者自行確認是否需要並執行）。
- `.env` 中含 API key，`EnvironmentFile` 路徑權限務必設為僅該使用者可讀
  （`chmod 600 .env`）。

## 5. 文件卡在 processing 狀態的處理 SOP

### 問題說明

LightRAG 文件管理通常有狀態機（如 `pending` → `processing` → `processed` /
`failed`）。若 LLM API 超時、進程被中斷，文件可能卡在 `processing`，
之後重新整理會被誤判為「正在處理中」而跳過，永遠不會完成。

### 建議做法

1. **先觀察，不要急著動手**：透過 LightRAG API 查詢文件狀態列表：
   ```bash
   curl -s http://localhost:9621/documents | python3 -m json.tool
   ```
   找出狀態為 `processing` 但已經很久沒更新（比對 `updated_at` 與目前時間）
   的文件。
2. **確認 server 是否仍在跑該文件**：檢查 server log / `journalctl`，看是否
   有對應的活動。若 server 已重啟過但狀態沒更新，代表狀態確實卡死。
3. **重置狀態**：
   - 若 API 有提供重新處理端點（如 `/documents/{id}/reprocess` 或刪除後
     重新 ingest），優先使用官方 API。實際端點名稱務必以
     `curl http://localhost:9621/openapi.json` 確認當前版本提供什麼。
   - 若無官方端點，需要直接編輯 `lightrag_workspace/` 中的文件狀態存儲
     （通常是 JSON/KV store），將該文件狀態改回 `pending`，**動手前先備份
     整個 `lightrag_workspace/`**。
4. **重新觸發處理**：重啟 server 或呼叫 scan/ingest 端點，讓狀態回到
   `pending` 的文件被重新撿起。
5. 處理完成後，跑一次 `golden_check.py` 確認沒有因為中途中斷造成圖譜不完整。

### 相關命令

```bash
# 備份 workspace 再動手
tar czf outputs/workspace_before_fix_$(date -u +%Y%m%dT%H%M%SZ).tar.gz lightrag_workspace/

# 查詢文件狀態
curl -s http://localhost:9621/documents | python3 -m json.tool | grep -A2 -i processing
```

### 風險

- 直接編輯 workspace 內部檔案是 unsupported 操作，版本不同格式可能不同，
  操作前務必備份；若不確定格式，優先選擇「重建」而非「手動編輯」。
- 重複處理同一份文件可能造成 entity/relation 重複，處理後務必跑
  `wiki_lint.py` 與抽樣查詢確認。

## 6. 備份要做還原演練（先解開驗證能否還原）

### 問題說明

「有備份」不等於「能還原」。常見失敗：tar 打包時路徑錯誤、漏掉關鍵檔案、
壓縮過程中斷產生壞檔，直到真正需要還原時才發現備份不可用。

### 建議做法

- 每次備份後**立即自動執行還原演練**：解壓到暫存目錄，檢查關鍵檔案
  （`wiki/index.md`、`schema/`、`lightrag_workspace/` 是否存在且非空）。
- 演練通過才視為備份有效（印出 `[OK]`），否則備份視為失敗，不留下
  半成品檔案。
- 實作見 `scripts/backup_lab.sh`（打包 + 自動演練）與
  `scripts/restore_lab.sh`（正式還原，預設也走暫存目錄演練模式，
  需 `--force` 才覆寫既有目錄）。

### 相關命令

```bash
bash scripts/backup_lab.sh
bash scripts/restore_lab.sh outputs/backups/lab_backup_<timestamp>.tar.gz
bash scripts/restore_lab.sh outputs/backups/lab_backup_<timestamp>.tar.gz --target ./restore_test --force
```

### 風險

- 備份不含 `.env`（依安全規則），還原後仍需手動依 `.env.example` 重建
  `.env`，這是預期行為，不是 bug。
- 備份不含 `lightrag_workspace/` 內的大型向量索引時間成本可能很高；
  若 workspace 過大，考慮改用 git-lfs 或獨立備份工具（見第 7 節）。

## 7. Git 衛生（排除 outputs/、git-lfs 或獨立備份）

### 問題說明

`outputs/`（含備份 tar.gz）與 `lightrag_workspace/`（向量索引、KV cache）
體積可能達數百 MB 到 GB 級，不適合進 git 一般物件儲存，會讓 repo 越來越肥、
clone 變慢。

### 建議做法

1. 確認 `.gitignore` 已排除：
   - `lightrag_workspace/`（已存在）
   - `outputs/`（已存在，含備份檔）
   - `.env`（已存在）
2. 若未來需要把備份納入版本控制（例如團隊共享），**不要**直接 `git add`
   大型 tar.gz，改用：
   - **Git LFS**：`git lfs track "outputs/backups/*.tar.gz"`，僅在團隊都
     已安裝 `git-lfs` 且了解 LFS 儲存成本時採用。
   - **獨立備份目標**：將 `outputs/backups/` 同步到外部儲存
     （雲端硬碟、NAS、物件儲存），repo 本身只保留備份「清單」
     （檔名 + checksum + 時間），不存實際內容。
3. 定期執行 `git status` 確認沒有意外把大型檔案加入追蹤；
   若已不小心 commit 過大型檔案，需要 `git filter-repo` 之類工具清理歷史
   ——這屬於 destructive 操作，**需先取得使用者明確同意**才能執行。

### 相關命令

```bash
cat .gitignore
git status
# 確認沒有大型檔案被追蹤
git ls-files -s | awk '{print $4}' | xargs -I{} du -h {} 2>/dev/null | sort -rh | head -10
```

### 風險

- Git LFS 有額外的儲存/流量配額限制（依託管平台），導入前需確認方案。
- 清理 git 歷史中已存在的大型檔案屬於高風險操作，務必先備份整個 repo
  並取得使用者同意。

## 8. 建立檢索品質的黃金問題集

### 問題說明

LightRAG 抽取與查詢的品質會隨著文件增加、模型更換而變動，需要一組固定的
「黃金問題」搭配預期關鍵詞/結構，作為迴歸測試基準。

### 建議做法

- 維護 `scripts/golden_questions.yml`：每題包含 `title` 與 `question`，
  涵蓋系統架構、儲存層、查詢模式、entity/relation、references 完整性、
  Wiki 編譯流程等核心面向（本次已擴充至 6 題以上，見該檔案）。
- 使用 `scripts/golden_check.py` 對每題執行查詢，檢查：
  - 是否回應非空、是否包含 references。
  - （可選）是否包含預期關鍵詞清單。
- 建議在每次「重大變更」後執行一次（換模型、批量匯入新文件、修改 prompt）：
  ```bash
  python scripts/golden_check.py
  ```
- 將每次執行結果（時間、模型版本、是否通過）記錄到
  `docs/IMPLEMENTATION_NOTES.md`，方便追蹤品質隨時間/版本的變化趨勢。

### 風險

- 黃金問題集的答案品質會隨知識庫內容增長而自然變化（更多文件 = 更豐富的
  回答），不應期望逐字相同，重點是「結構完整性」（有 references、
  有回應、沒有錯誤）而非「文字完全一致」。

## 9. PostgreSQL 評估

### 問題說明

LightRAG 預設使用內建的檔案型 KV/向量/圖儲存（位於 `lightrag_workspace/`）。
PostgreSQL（搭配 pgvector / Apache AGE 等擴充）可作為替代儲存後端，
提供多進程併發存取、更好的查詢效能與正式環境的備份/複寫能力。

### 何時該升級的判斷準則

升級到 PostgreSQL，當以下任一條件成立：

- 需要**多個進程/服務同時存取**同一份知識庫（檔案型儲存在併發寫入下
  容易鎖死或損毀）。
- `lightrag_workspace/` 中單一 JSON/KV 檔案已成長到難以備份/還原的大小
  （例如 > 數百 MB，`backup_lab.sh` 還原演練時間明顯變長）。
- 需要對接現有的資料庫維運體系（既有 PostgreSQL 監控、備份排程、
  HA 機制）。
- 需要更複雜的圖查詢（多 hop 關係查詢）且檔案型 graph store 效能不足。

### 第一版是否採用

**第一版不採用**。理由：

- 第一版文件量小，檔案型儲存的效能與容量都足夠。
- 引入 PostgreSQL 需要額外的服務常駐化、連線設定、權限管理，
  增加環境複雜度，與第一版「先求穩、可驗證」的目標不符。
- 保留 `.env` 中的 binding 設定為未來切換預留空間即可（無需現在安裝）。

### 相關命令（未來評估用）

```bash
# 確認 LightRAG 是否支援 postgres storage backend（依官方文件/--help 確認）
lightrag-server --help | grep -i -E "postgres|storage"
```

### 風險

- 切換儲存後端通常**不是無痛遷移**，舊資料可能需要重新 ingest，
  切換前務必完整備份並評估重新匯入成本（見第 2 節）。

## 10. Rerank 評估

### 問題說明

`.env.example` 中 `RERANK_BINDING=null`、`RERANK_BY_DEFAULT=false`，目前查詢
結果完全依賴 embedding 相似度排序。Rerank 模型（如 cross-encoder）能在
embedding 初篩後做二次精排，提升 top-k 結果的相關性，但會增加查詢延遲與
額外的模型推理成本。

### 何時該升級的判斷準則

啟用 Rerank，當：

- `golden_check.py` 顯示查詢結果中**相關內容排序靠後**（答案存在於
  reference 中，但不在前幾筆，導致 LLM 生成答案時被截斷的 context 忽略）。
- 知識庫規模變大（chunk 數量明顯增加），embedding-only 排序的精確度
  下降變明顯。
- 已有可用的本地 rerank 模型（例如透過 Ollama 或其他 OpenAI-compatible
  endpoint 提供 rerank API），且確認延遲增加在可接受範圍。

### 第一版是否採用

**第一版不採用**，但設定保留為「已預留、未啟用」（`.env.example` 現狀即可）。
理由：

- 文件量小時，embedding 排序已足夠分辨相關性，rerank 帶來的提升有限。
- 先建立黃金問題集（第 8 節）並觀察基準表現，未來若發現排序問題，
  再針對性開啟 rerank 並用黃金問題集做 A/B 比較，比一開始就開啟更容易
  量化效益。

### 相關命令（未來評估用）

```bash
# 開啟前後分別跑黃金問題集，比較 references 排序與回答品質
RERANK_BY_DEFAULT=false python scripts/golden_check.py > outputs/golden_no_rerank.json
RERANK_BY_DEFAULT=true  python scripts/golden_check.py > outputs/golden_with_rerank.json
```

### 風險

- Rerank 模型本身也需要資源（記憶體/推理時間），若用遠端 API 還會增加
  每次查詢的延遲與費用，需與第 2 節成本估算一併考量。

## 11. MCP 化評估

### 問題說明

將 `query_lightrag.py` / `compile_wiki_page.py` 等查詢與 Wiki 編譯能力，
包裝成 MCP server，讓 Agent（Claude Code 或其他 MCP client）可以直接呼叫，
而不需要每次都手動跑腳本。

### 何時該升級的判斷準則

導入 MCP 化，當：

- Wiki 與 LightRAG 的查詢流程已穩定（golden 問題集穩定通過），
  不再需要頻繁手動調整腳本參數。
- 有明確的 Agent 工作流需要「先查 Wiki，不足再查 LightRAG，必要時寫回
  Wiki」的自動化迴路（對應 CLAUDE.md 核心架構中 Agent/MCP 那一層）。
- 團隊中有多個 Agent/工具需要共用同一套查詢能力，重複維護多份呼叫腳本
  成本變高。

### 第一版是否採用

**第一版不採用，但建議規劃為下一階段的第一個任務**。理由：

- 第一版重點是把 ingest → wiki 編譯 → 驗收 的核心流程跑通並驗證品質
  （golden questions、wiki_lint）。
- MCP server 的封裝屬於「介面層」工作，建立在核心流程穩定之上才有意義，
  過早封裝會在核心邏輯仍變動時頻繁修改介面，浪費工作量。
- 規劃：未來可將 `scripts/query_lightrag.py`、
  `scripts/compile_wiki_page.py`、`scripts/wiki_lint.py` 的核心函式抽出，
  包裝為 MCP server 的 tools（例如 `query_wiki`、`query_lightrag`、
  `compile_wiki_page`），對應 CLAUDE.md 中「Agent / MCP」層。

### 風險

- MCP server 若直接暴露「寫入 wiki」能力給 Agent，需要明確的權限與審核
  機制，避免 Agent 自動寫入未經驗證的內容（違反可驗證性原則）。
